# 规则与函数管理 — 完整设计

> 本体中心 · RBox 层 · 双入口 + 注册中心方案

## 1. 设计定位

规则与函数是本体平台的业务逻辑层（RBox），既是本体语义的组成部分，也是可执行的业务引擎。

**核心定位**：规则和函数做好"积木块"，让技能层来"搭积木"。

```
本体模型层 (TBox): 实体 / 属性 / 关系
        ↓ 提供语义基础
规则与函数层 (RBox): 条件判断 / 计算逻辑
        ↓ 被引用和调用
技能层 (Skill) → Agent → 场景执行
```

## 2. 整体架构

### 2.1 三个入口/视图

| 入口 | 位置 | 职责 |
|------|------|------|
| 实体入口 | 实体详情 → "逻辑" Tab | 快速为当前实体创建/查看规则和函数 |
| 全局入口 | 本体中心 → 规则与函数 | 全局浏览、搜索、完整构建能力 |
| 注册中心 | 技能编辑器内嵌面板 | 按实体/类型/关键词检索，选取引用 |

### 2.2 核心交互流

- **建模人员**：编辑实体 → 逻辑 Tab → 新建规则/函数 → 自动绑定当前实体
- **规则分析师**：全局管理页 → 新建/编辑 → 选择关联实体 → 可视化构建条件
- **技能开发者**：编辑技能 → 打开注册中心面板 → 搜索 → 选中规则/函数 → 自动注入引用

### 2.3 关联关系

- 规则条件中可调用函数作为计算单元
- 规则触发时引用行动（行动管理模块）执行
- 技能可引用规则和函数作为 tool

## 3. 数据模型变更

### 3.1 BusinessRule — 增加字段

| 新字段 | 类型 | 说明 |
|--------|------|------|
| `description` | Text | 规则的业务描述，供注册中心展示 |
| `tags` | JSON (list) | 分类标签，如 `["预警", "用户"]` |
| `input_params` | JSON | 执行所需的外部输入参数定义 |
| `output_schema` | JSON | 执行结果的结构描述 |

### 3.2 OntologyFunction — 增加字段

| 新字段 | 类型 | 说明 |
|--------|------|------|
| `tags` | JSON (list) | 分类标签 |
| `callable_name` | String(100) | 注册名称，供规则条件和技能引用 |

### 3.3 conditions_json 扩展 — 函数调用类型

现有格式:
```json
{"field": "Entity.property", "operator": ">=", "value": 0.7}
```

新增函数调用类型:
```json
{
  "type": "function_call",
  "function_id": "xxx-xxx",
  "callable_name": "calc_churn_score",
  "params": {"user_id": "$context.user_id"},
  "operator": ">=",
  "value": 0.7,
  "display": "流失评分 ≥ 0.7"
}
```

### 3.4 Skill.tools 扩展

```json
{
  "tools": [
    {"type": "rule", "rule_id": "xxx", "name": "高风险用户识别", "invoke_mode": "evaluate"},
    {"type": "function", "function_id": "yyy", "callable_name": "calc_churn_score", "invoke_mode": "call"},
    {"type": "code", "skill_tool_id": "zzz", "name": "send_notification"}
  ]
}
```

## 4. 规则构建器

### 4.1 形态

右侧抽屉，width 640px（替代原设计的 560px 弹窗）。

### 4.2 表单结构

- **基本信息**：规则名称、描述、关联实体（实体入口时锁定）、优先级、标签
- **触发条件**：支持结构化模式和表达式模式切换
- **关联行动**：从行动管理中选择，配置参数映射
- **输入参数定义**：声明规则执行所需的外部输入

### 4.3 结构化条件模式

字段选择器分两组：
- **实体属性组**：列出关联实体的所有属性
- **函数调用组**：列出可用函数，选中后展开参数填写区

操作符根据字段类型自动过滤：
- 数值型：`>`, `>=`, `<`, `<=`, `==`, `!=`
- 字符串型：`==`, `!=`, `in`, `not_in`
- 布尔型：`==`（是/否）
- 时间型：`>=`, `<=`（配合时间偏移 `NOW()-Nd`）

条件组：
- 默认 AND
- 支持添加 OR 条件组（组内 AND，组间 OR）
- 对应 `match_mode`: `all` / `at_least_N`

### 4.4 表达式条件模式

代码编辑器，支持：
- 自动补全：实体属性、`$fn.` 函数列表
- 语法高亮
- 右侧预览解析后的结构化 JSON

两种模式可互相切换（表达式→结构化解析失败时提示留在表达式模式）。

## 5. 函数构建器

### 5.1 形态

右侧抽屉，width 640px。

### 5.2 表单结构

- **基本信息**：函数名称、调用名称（callable_name，自动生成）、描述、关联实体（可选）、返回类型、是否派生属性、标签
- **输入参数**：参数名、类型、必填、说明
- **逻辑体**：根据逻辑类型切换编辑器
- **测试面板**：输入参数 JSON → 运行 → 展示结果和耗时

### 5.3 三种逻辑类型

| 逻辑类型 | 编辑器 | 自动补全 | 适用场景 |
|----------|--------|----------|----------|
| 表达式 | 轻量高亮输入 | 实体属性、内置数学/字符串函数 | 简单计算 |
| SQL | SQL 编辑器 | 表名、字段名、SQL 关键字 | 聚合查询 |
| Python | 代码编辑器 | 参数名、白名单库 | 复杂逻辑 |

### 5.4 派生属性特殊处理

开启后：
- 关联实体变为必填
- 函数作为实体的虚拟属性出现在属性列表中
- 规则构建器字段选择器中可直接看到（归在"计算属性"分组）
- 调用名称自动加前缀：`entity_name.callable_name`

## 6. 注册中心与技能集成

### 6.1 注册中心面板

嵌入技能编辑器，通过"添加能力"按钮打开：
- 按实体分组展示所有规则和函数
- 每项显示：名称、描述、输入参数、输出类型
- 支持搜索和按类型筛选（规则/函数/全部）
- 点击"+ 添加"自动写入技能的 tools 配置

### 6.2 技能执行时的调用链路

```
Skill 执行
  → tool call: evaluate_rule / call_function
  → skill_executor 路由:
      evaluate_rule → RuleEvaluator.evaluate(rule, user_id)
      call_function → FunctionExecutor.execute(function, params)
  → 返回结果
```

`FunctionExecutor`（新增服务）根据 logic_type 分发：
- expression → 表达式求值
- sql → 注入表名后执行
- python → skill_sandbox.py 沙箱执行

### 6.3 反向查询

规则/函数详情页展示"被引用情况"：
- 查询 Skill.tools 中包含该 rule_id/function_id 的记录
- 查询 BusinessRule.conditions_json 中引用该 function_id 的记录

## 7. 实体入口 — "逻辑" Tab

### 7.1 布局

实体详情页新增"逻辑"Tab，分三个区域：
- **规则**：该实体绑定的所有规则，简要列表
- **函数**：该实体绑定的所有函数
- **行动**：该实体绑定的所有行动

每个区域右上角有"+ 新建"按钮，打开构建器时 entity_id 预填锁定。

### 7.2 与全局管理页的关系

| 维度 | 实体入口 | 全局入口 |
|------|----------|----------|
| 范围 | 只看当前实体 | 所有实体 + 独立函数 |
| 创建 | entity_id 锁定 | entity_id 手选 |
| 筛选 | 不需要按实体筛选 | 支持按实体、标签、状态筛选 |
| 适合谁 | 本体建模人员 | 业务规则分析师 |

## 8. 全局管理页

### 8.1 页面结构

- Tab 切换：规则管理 / 函数管理
- 统计栏：4 个统计卡片
- 工具栏：搜索 + 实体筛选 + 标签筛选 + 状态筛选 + 新建按钮
- 列表：可展开卡片，收起态展示摘要，展开态展示完整详情和操作

### 8.2 规则 Tab 统计

总规则数 / 活跃规则 / 需关注 / 已禁用

### 8.3 函数 Tab 统计

总函数数 / 已激活 / 派生属性 / 独立函数

### 8.4 列表卡片信息

规则卡片收起态：状态圆点 + 规则名 + 所属实体 + 标签 + 优先级 + 触发次数 + 被引用数

函数卡片收起态：函数名 + callable_name + 所属实体 + 逻辑类型标签 + 派生/独立标签 + 调用次数

### 8.5 筛选器

- 实体筛选：下拉，列出有规则/函数的实体 + "独立函数"
- 标签筛选：从现有 tags 聚合，多选
- 被引用筛选：`[全部] [被技能引用] [未引用]`
- 状态/优先级/逻辑类型：根据当前 Tab 显示对应筛选

### 8.6 批量操作

多选后顶部出现操作栏：批量启用/禁用、批量打标签、批量删除（二次确认）

## 9. 新增后端服务

| 服务 | 职责 |
|------|------|
| `FunctionExecutor` | 根据 logic_type 执行函数（表达式/SQL/Python），统一入口 |
| 注册中心 API | 提供聚合查询接口：按实体分组、按标签筛选、返回元信息 |
| 引用查询 API | 查询某规则/函数被哪些技能和规则引用 |

## 10. 不做的事

- 不做图形化拖拽流程编排（复杂编排由技能层负责）
- 不做函数组合函数（函数链）
- 不做规则链式触发（已有 evaluate_all 的优先级互斥机制足够）
- 不做独立的"注册中心表"（规则/函数本身就是数据源）
